Turning Architecture into a System with Power Apps, SharePoint, and LeanIX

Enterprise architecture • Operating model • Power Apps • SharePoint • LeanIX

Architecture becomes much more useful when it moves beyond static documents. A practical internal setup combines reusable templates, lightweight workflow, and relationship management so decisions, reviews, and architecture assets can actually be used by delivery teams.

The problem with document-only architecture

Many architecture teams produce good content, but the content lives in disconnected documents, slide decks, or folders. That makes it hard to track ownership, hard to reuse templates, and hard to connect reviews or decisions back to the systems and capabilities they affect.

A practical operating model

  • SharePoint Lists: store core records such as reviews, decisions, and architecture overviews.
  • Power Apps: provide an intake and workflow layer so teams can submit requests and update records without editing raw lists.
  • LeanIX: maintain relationships between applications, capabilities, initiatives, and technology objects.
  • Architecture repository: publish templates, guidance, and reference material so the operating model stays understandable.

How the templates fit

Architecture Overview

Use as the intake and framing record for new work. This can start as markdown and later become a SharePoint List item backed by a Power Apps form.

Decision Record

Store meaningful decisions in a list that supports filtering, ownership, and review triggers. Link those decisions to affected systems in LeanIX.

Review Note

Use one structure for ARB, TRB, design review, and exception review. A simple review type field lets the same template support multiple governance motions.

Current vs Target State

Use for roadmap framing and transition planning, with major application and capability relationships maintained in LeanIX.

Example workflow

  1. A team submits an architecture request through a Power Apps form.
  2. The request writes to a SharePoint List using the architecture overview structure.
  3. If the request needs governance, an ARB or TRB review note is created using the same review template.
  4. Important decisions are captured in a decision record with status, owner, and review trigger.
  5. Applications, capabilities, and related initiatives are linked in LeanIX so the decision is connected to the broader estate.

Why this works

This approach keeps the system lightweight. SharePoint and Power Apps handle workflow and usability. LeanIX handles architecture relationships. The repository stays focused on templates, guidance, and examples. Together they create an architecture operating system instead of another pile of disconnected documents.
