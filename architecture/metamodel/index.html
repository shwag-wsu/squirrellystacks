Architecture Metamodel

The metamodel is the shared language behind the rest of the repository. It defines what we describe, how artifacts connect, and which viewpoints are useful enough to keep reusing.

Why this matters

Without a consistent metamodel, architecture content becomes a pile of diagrams and documents that use the same words differently. The practical goal here is simpler: make sure capability maps, application inventories, reference architectures, and governance records are describing the same core objects.

Core architecture objects

  • Capabilities: what the enterprise needs to be able to do.
  • Business processes: how work is carried out across roles and teams.
  • Applications and services: the systems that support or automate work.
  • Data entities: the information that must stay trustworthy across systems.
  • Technology platforms: the infrastructure and enabling tools underneath applications.

Reusable viewpoints

Planning viewpoint

Connect business outcomes, capabilities, and roadmap options so investment conversations stay outcome-oriented.

Delivery viewpoint

Show solution boundaries, integration points, and key data flows so implementation teams know where responsibility sits.

Governance viewpoint

Capture standards, exceptions, and decision rationale so review outcomes stay traceable after the meeting ends.
